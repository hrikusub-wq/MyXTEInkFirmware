# AI間ハンドオフ(Claude ⇄ Antigravity 分担作業)

## なぜこのスキルが必要か

このリポジトリでは、ClaudeとAntigravityが同じソースツリーを分担して編集することがある。あるセッションで、Claudeが実装した「AutoStandbyの5分タイマー」「ホーム画面限定の監視範囲」が、Antigravity側が**未コミットの変更ごとファイルを上書き**した(前セッションの消えたコードを復元するためのgit checkout)ことで一時的に失われる事故が実際に起きた。原因はシンプルで、**Claudeが実装した内容がまだコミットされていなかった**ため、Gitの履歴に残らず復元の手がかりが無かったこと。

この事故は「担当を代わる前に一度コミットしておく」だけでほぼ防げる。以下の手順を徹底する。

## 手順1: セッション開始時 — 制約事項と直前の状態を確認する

1. リポジトリ直下に `AI_RULES.md` があれば必ず読む。ESP32-C3のRAM制約・E-inkパネルの電源管理など、事前に知らないとクラッシュやバッテリー消費バグを踏みやすい落とし穴がまとまっている。無ければ、作業内容がハードウェア制約に関わりそうな場合(メモリ確保、電源管理、割り込み処理など)は作成を提案する。
2. `Antigravity/` フォルダ(無ければ他に引き継ぎノートが置かれていそうな場所)に、自分宛て、あるいは直近の引き継ぎノートが無いか確認する。あれば読む。
3. **引き継ぎノートに書かれている内容を鵜呑みにしない。** 「〇〇を実装した」「〇〇は解決済み」といった記述は、実際に該当ファイルを読む/`git diff`を取るなどして裏取りしてから前提にする。ノートの記述と実コードが食い違っていることは珍しくない(伝聞情報や、その後さらに他のAIが上書きした結果ズレている場合がある)。このセッションでも、引き継ぎノートには書かれていなかった「電源キー短押しでBluetooth起動」機能が実際には実装済みだったことや、自分の直前の実装が巻き戻っていたことが`git diff`で判明した。
4. `git status`と`git log`で、自分が把握していないコミットや変更が無いか確認する。想定と違う変更があれば、それを消したり上書きしたりする前に、まず内容を理解する(このリポジトリのグローバルなGit運用ルールと同じ: 不明な変更は保護してから作業する)。

## 手順2: 作業の区切り・担当交代の前 — 必ずコミットする

**他のAI(またはユーザー)がファイルを操作する可能性がある区切り目では、その都度コミットする。** 具体的には:

* ひとまとまりの機能を実装し終えて、ビルド確認まで終わったとき
* ユーザーが「Antigravityに任せます」「ここまでで一旦切ります」のように担当交代を示唆したとき
* 長めの調査・議論に入る前(コード変更が既に手元にある状態で会話が長引くと、その間に他のAIが同じファイルを触るリスクが上がる)

コミットメッセージは完成度を問わない。作業途中でも `WIP: ` 接頭辞を付けてコミットしてよい(スカッシュや整理は後でいくらでもできるが、コミットされていない変更は他のAIのファイル操作で消える可能性がある)。

```bash
git add -A
git commit -m "WIP: <何をした/どこまで終わったかの短い説明>"
```

**必ずユーザーに確認してから実行すること。** 通常のコミット同様、無断でコミットしない。ユーザーから「担当を代わる」「区切りをつけたい」という意図が読み取れたら、「ここで一度コミットしておきましょうか?」と提案する。

## 手順3: 担当交代時 — 引き継ぎノートを書く

作業を他のAIに引き継ぐとき(ユーザーがそう指示したとき、または区切りの良いタイミングで提案するとき)は、`Antigravity/`フォルダに引き継ぎノートを書く。命名規則:

```
Antigravity/report_for_<引き継ぎ先>_<YYYYMMDD>_<内容を表す短い単語>.md
```

例: `Antigravity/report_for_antigravity_20260721_deep_sleep_timeout.md`

構成の目安(このリポジトリの過去の引き継ぎノートと同じ形式):

```markdown
# <タイトル>

この報告書の目的(誰が何のために読むか)を1行で。

## 1. 今回やったこと(スコープ内)
- ファイルパス・関数名・実装内容を具体的に

## 2. スコープ外だが気づいた/直した内容(あれば)

## 3. 次のAIへの引き継ぎ事項
- 実機確認がまだ/ビルド確認のみ、など「検証していないこと」を明記する
- ハードウェア制約や設計上の理由で「これはやってはいけない」ことがあれば書く
- 未解決のTODO・ユーザーへの未回答事項があれば書く
```

**「実装した」と書く内容は、実際にファイルを保存・ビルド確認した後に書く。** 会話の中で「〜する予定」と言っただけの内容を実装済みとして書かない(手順1で述べた「ノートを鵜呑みにしない」問題を自分が発生させないため)。

## この3手順の関係

```
[セッション開始] → 手順1: AI_RULES.md・引き継ぎノートを読む(裏取りしてから信じる)
      ↓
   作業する
      ↓
[区切り/交代の前] → 手順2: コミットする(ユーザーに確認してから)
      ↓
[担当を渡す]      → 手順3: 引き継ぎノートを書く
```

ユーザーから「コミットして、引き継ぎノートを書いて」と一言指示されたときは、手順2→3をまとめて実行すればよい。逆に「Antigravityから引き継いだ」「Antigravityと分担した続きをやる」と言われたときは、手順1から始める。
